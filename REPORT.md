# 데이터 수집 가능성 조사 보고서

> 작성일: 2026-05-09
> 담당: 데이터 수집 파트

---

## 1. 목적

AR 부동산 분석 서비스에 필요한 **6개 외부 API**가 실제로 동작하는지, 데이터가 서로 연결 가능한지 검증.

---

## 2. 서비스 흐름과 각 API의 역할

사용자가 건물을 카메라로 촬영하면 아래 4단계를 거쳐 AR 화면에 정보가 표시됩니다.

```
Phase 1 (Input)       스마트폰 GPS + IMU 센서 + 카메라 촬영
        ↓
Phase 2 (Processing)  건물 식별 — 어떤 건물을 보고 있는가?
        ↓
Phase 3 (Enrichment)  건물 정보 조회 — 시세/법적위험/ESG
        ↓
Phase 4 (Output)      AR 오버레이로 Data Card 출력
```

### Phase 2에서 필요한 API

| API | 역할 | 왜 필요한가 |
|-----|------|-----------|
| **Vworld 2D 데이터 API** | 현재 위치 반경 500m 내 건물의 **폴리곤(옥상 테두리 좌표)** 을 GeoJSON으로 수신 | 센서로 계산한 카메라 방향 벡터(Ray)가 어느 건물 폴리곤과 충돌하는지 **레이 캐스팅**으로 판별해야 함. 이 폴리곤이 없으면 건물 식별 자체가 불가능. |
| **카카오맵 API** | 좌표 ↔ 주소 변환, POI(건물명) 검색 | 레이 캐스팅으로 찾은 건물의 주소를 확인하거나, BD_MGT_SN에서 추출한 도로명주소를 교차 검증하는 **보조 수단**. |
| **Google Cloud Vision API** | 카메라에 찍힌 건물 간판의 **텍스트(OCR)** 추출 | 간판에서 "101동", "푸르지오" 같은 텍스트가 포착되면, 레이 캐스팅 후보군 중에서 최종 건물을 확정짓는 **교차 검증 도구**. 간판이 없거나 야간이면 OCR은 실패할 수 있으므로 메인이 아닌 보정 역할. |

### Phase 3에서 필요한 API

건물이 식별되면 **건물 고유 ID(PNU)** 를 기반으로 공공데이터를 조회합니다.

| API | 역할 | AR 화면에 표시될 정보 |
|-----|------|-------------------|
| **국토교통부 실거래가 API** | 해당 건물의 최근 매매/전월세 거래 내역 조회 | **예상 가격: 11.5억 KRW**, 전용면적 84m², 최근 거래일 등 → 경제적 가치 판단 |
| **건축물대장 API** | 건물의 법적 정보 조회 (용도, 사용승인일, 위반건축물 여부) | **법적 리스크: 낮음/높음**, 위반건축물 이력, 건폐율/용적률 등 → 리걸테크 지표 |
| **건물에너지사용량 API** | 건물의 월별 전기/가스 사용량 조회 | **에너지 등급: A+**, 탄소 배출량 XX kg CO₂ 등 → ESG 지표 |

### 한 줄 요약

> Vworld 폴리곤으로 **"어떤 건물인지"** 식별하고, 실거래가/건축물대장/에너지로 **"그 건물의 시세·법적위험·ESG"** 를 보여주는 것이 핵심 데이터 파이프라인.

#### 건축물대장 API (건축HUB)

| 항목 | 결과 |
|------|------|
| 엔드포인트 | `https://apis.data.go.kr/1613000/BldRgstHubService/getBrTitleInfo` |
| 인증 | 공공데이터포털 serviceKey (즉시 승인) |

| 필드 | 설명 | 샘플 값 |

| `bldNm` | 건물명 | 청운벽산빌리지 |
| `mainPurpsCdNm` | 주용도 | 공동주택 |
| `totArea` | 연면적 (m²) | 5489.43 |
| `grndFlrCnt` | 지상층수 | 3 |
| `ugrndFlrCnt` | 지하층수 | 2 |
| `useAprDay` | 사용승인일 | 19881111 |
| `dongNm` | 동명칭 | 9동 |
| `bylotCnt` | 위반건축물 수 | 0 |
| `strctCdNm` | 구조 | 철근콘크리트구조 |

**집합건물(아파트) 1:N 문제**: 1필지에 10개동 (1동~10동, 상가동) 존재 → 동 이름 텍스트 매칭 필요 (~90% 매칭률)

**해결된 이슈**: Python urllib의 `Accept-Encoding: identity` 헤더로 인해 data.go.kr HTTPS 서버가 빈 응답(Content-Length: 0)을 반환하는 문제. `req.add_unredirected_header("Accept-Encoding", "")` 추가로 해결.

## 4. 핵심 발견: 데이터 연계 방법

### 용어 설명

| 용어 | 정식 명칭 | 설명 |
|------|---------|------|
| **BD_MGT_SN** | 건물관리번호 (25자리) | 행정안전부가 건물마다 부여한 고유 식별번호. Vworld API 응답에 포함됨. 예: `1141011300100120000000001` |
| **PNU** | 필지고유번호 (19자리) | 토지(필지) 단위의 고유번호. **Vworld에서는 별도 필드 없이 `bd_mgt_sn[:19]`로 추출.** 시도+시군구+읍면동+리+토지구분+본번+부번으로 구성. 예: `1111010100100010000` |
| **LAWD_CD** | 법정동코드 (5자리) | 시도코드(2) + 시군구코드(3). 실거래가 API의 검색 키. 예: `11110` (서울 종로구) |

### BD_MGT_SN은 공통키가 아니다

프로젝트 초기에는 Vworld에서 받은 BD_MGT_SN(건물관리번호)으로 실거래가, 건축물대장, 에너지 API를 모두 조회할 수 있다고 가정했지만, **실제로는 불가능**.

실거래가, 건축물대장, 에너지 API 모두 BD_MGT_SN을 검색 키로 받지 않음. 대신 **PNU(필지고유번호)를 분해**하여 각 API에 맞는 파라미터로 변환해야 함.

### 실제 연계 방법: PNU 분해

```
Vworld에서 건물 조회 → bd_mgt_sn 획득 → bd_mgt_sn[:19] = PNU
    │
    └── PNU (19자리) 분해:
        ├── 시군구코드 (5자리) ──→ 실거래가 API의 LAWD_CD
        ├── 법정동코드 (5자리) ──→ 건축물대장/에너지 API의 bjdongCd
        ├── 본번 (4자리)       ──→ bun
        └── 부번 (4자리)       ──→ ji
```

### PNU 구조 (19자리)

```
[시도 2자리][시군구 3자리][읍면동 3자리][리 2자리][토지구분 1자리][본번 4자리][부번 4자리]

예시: 1111010100100010000
  시도     = 11    (서울특별시)
  시군구    = 110   (종로구)
  읍면동    = 101   (청운동)
  리       = 00
  토지구분  = 1     (대지)
  본번     = 0001
  부번     = 0000

→ 실거래가 조회 시: LAWD_CD = "11110" (시도+시군구)
→ 건축물대장 조회 시: sigunguCd = "11110", bjdongCd = "10100", bun = "0001", ji = "0000"
```

PNU 파싱 로직은 `api_test/config.py`의 `parse_pnu()` 함수로 구현 완료.

## 5. 리스크 정리

| 리스크 | 심각도 | 대응 |
|--------|:------:|------|
| BD_MGT_SN이 공통키로 안 됨 | **중** | PNU 기반 연계로 전환 (구현 완료) |
| 아파트 1필지 다동 매칭 실패 | **중** | 동 이름 텍스트 비교 (~90% 매칭률) |
| ~~건축HUB API 엔드포인트 불명확~~ | ~~중~~ | **해결** — Accept-Encoding 헤더 수정 + 에너지 API 파라미터 수정 |
| API 일일 1,000회 제한 | 저 | 데모 수준에선 충분. 캐싱 도입 검토 |
| Vworld 3D API 폐쇄 | 저 | 2D 폴리곤으로 충분 (옥상 윤곽) |
| 에너지 데이터 커버리지 | 저~중 | 소규모 건물 / 일부 주소에서 데이터 없을 수 있음 확인됨 |
