# AR 스마트 부동산 분석 플랫폼

> 투빅스 컨퍼런스 프로젝트
> 주제: 카메라 기반 AR 부동산 분석 서비스 (멀티모달: Vision + Sensor + Tabular)

## 데이터 수집 가능성 조사

사용자가 건물을 카메라로 촬영하면 센서(GPS+IMU) + 비전(OCR) + 공공데이터(시세/법적정보/ESG)를 결합해 정보를 AR로 보여주는 서비스.

**조사의 목적**: 각 API가 실제로 동작하는지, 데이터가 서로 연결 가능한지 확인.

---

## 서비스 흐름과 각 API의 역할

```
Phase 1 (Input)       스마트폰 GPS + IMU 센서 + 카메라 촬영
        ↓
Phase 2 (Processing)  건물 식별 — 어떤 건물을 보고 있는가?
        ↓
Phase 3 (Enrichment)  건물 정보 조회 — 시세/법적위험/ESG
        ↓
Phase 4 (Output)      AR 오버레이로 Data Card 출력
```

### Phase 2: 건물 식별에 필요한 API

| API | 역할 | 왜 필요한가 |
|-----|------|-----------|
| **Vworld 2D 데이터 API** | 현재 위치 반경 500m 내 건물의 **폴리곤(옥상 테두리 좌표)** 을 GeoJSON으로 수신 | 센서로 계산한 카메라 방향 벡터(Ray)가 어느 건물 폴리곤과 충돌하는지 **레이 캐스팅**으로 판별. 이 폴리곤이 없으면 건물 식별 자체가 불가능. |
| **카카오맵 API** | 좌표 ↔ 주소 변환, POI(건물명) 검색 | 레이 캐스팅으로 찾은 건물의 주소를 확인하거나, 도로명주소를 교차 검증하는 **보조 수단**. |
| **Google Cloud Vision API** | 카메라에 찍힌 건물 간판의 **텍스트(OCR)** 추출 | 간판에서 "101동", "푸르지오" 같은 텍스트가 포착되면 레이 캐스팅 후보군에서 최종 건물을 확정짓는 **교차 검증 도구**. |

### Phase 3: 건물 정보 조회에 필요한 API

건물이 식별되면 **PNU(필지고유번호)** 를 기반으로 공공데이터를 조회.

| API | 역할 | AR 화면에 표시될 정보 |
|-----|------|-------------------|
| **실거래가 API** | 해당 건물의 최근 매매/전월세 거래 내역 | **예상 가격: 11.5억 KRW**, 전용면적, 최근 거래일 등 |
| **건축물대장 API** | 건물의 법적 정보 (용도, 위반건축물 여부 등) | **법적 리스크: 낮음/높음**, 건폐율/용적률 등 |
| **건물에너지사용량 API** | 건물의 월별 전기/가스 사용량 | **에너지 등급: A+**, 탄소 배출량 등 |

## 용어 설명

| 용어 | 정식 명칭 | 설명 |
|------|---------|------|
| **BD_MGT_SN** | 건물관리번호 (25자리) | 행정안전부가 건물마다 부여한 고유 식별번호. Vworld API 응답에 포함. 예: `1141011300100120000000001` |
| **PNU** | 필지고유번호 (19자리) | 토지(필지) 단위의 고유번호. 시도+시군구+읍면동+리+토지구분+본번+부번. 예: `1111010100100010000` |
| **LAWD_CD** | 법정동코드 (5자리) | 시도코드(2) + 시군구코드(3). 실거래가 API의 검색 키. 예: `11110` (서울 종로구) |

## 조사 플랜 & 체크리스트

### Step 1: Vworld GIS건물통합정보 API (최우선)

| 항목 | 상태 |
|------|:----:|
| API 키 발급 (무료, vworld.kr) | **발급 완료** |
| GeoJSON 형태로 폴리곤 좌표 반환 | **대기중** (서비스URL 반영 대기) |
| BD_MGT_SN (25자리 건물관리번호) 포함 | ⬜ |
| PNU (19자리 필지고유번호) 포함 | ⬜ |
| 반경 기반 검색 (geomFilter + buffer) | ⬜ |
| 일일 호출 제한 확인 | 무제한 (마이페이지 확인) |

- **엔드포인트**: `http://api.vworld.kr/req/data`
- **레이어 후보**: LT_C_AISRESC, LT_C_SPBD, LT_C_UISRESC
- **리스크**: 3D 데이터 API는 2019년 폐쇄됨. 2D만 사용 가능.
- **이슈**: 서비스URL을 `http://localhost`로 변경 후 반영 대기중. 안 되면 키 재발급 필요.

### Step 2: 국토교통부 실거래가 API

| 항목 | 상태 |
|------|:----:|
| API 키 발급 + 승인 (data.go.kr) | **승인 완료** (즉시) |
| XML 응답 정상 확인 | **PASS** |
| 거래금액, 전용면적, 아파트명, 층, 건축년도 필드 | **PASS** (전체 32개 필드 확인) |
| BD_MGT_SN **미포함** 확인 | **PASS** (예상대로 미포함) |
| PNU → LAWD_CD 연계 | **PASS** |
| 일일 1,000회 제한 확인 | ⬜ |

- **엔드포인트**: `http://apis.data.go.kr/1613000/RTMSDataSvcAptTradeDev/getRTMSDataSvcAptTradeDev`
- **연계**: PNU → 시군구코드(5자리) + 법정동코드(5자리) + 지번 → 실거래가 조회
- **테스트 결과**: 종로구(11110) 2025년 1월 → 26건 반환. 인왕산아이파크 11.5억, 84.858m², 1층 등 확인.
- **확인된 필드**: `dealAmount`, `excluUseAr`, `aptNm`, `floor`, `buildYear`, `umdNm`, `jibun`, `bonbun`, `bubun`, `roadNm`, `sggCd`, `umdCd` 등
- **참고**: 전월세 API(`RTMSDataSvcAptRent`)는 별도 활용신청 필요 (현재 미신청 → 403)

### Step 3: 건축물대장 API

| 항목 | 상태 |
|------|:----:|
| API 키 발급 + 승인 | **승인 완료** (즉시) |
| mgmBldrgstPk (건축물대장 PK) 확인 | **대기중** (API 빈 응답) |
| BD_MGT_SN **미포함** 확인 | ⬜ |
| 위반건축물 이력 조회 가능 여부 | ⬜ |
| 아파트(집합건물) 1필지 다동 → 1:N 매칭 문제 | ⬜ |
| 일일 1,000회 제한 확인 | ⬜ |

- **엔드포인트**: `https://apis.data.go.kr/1613000/BldRgstHubService/getBrTitleInfo`
- **핵심 리스크**: 건축물대장 PK(mgmBldrgstPk) ≠ BD_MGT_SN. 아파트는 PNU만으로 특정 동 식별 불가 → 동 이름 텍스트 매칭 필요 (~90% 매칭률)
- **이슈**: 승인 완료 상태지만 HTTP 200 + 빈 응답 반환. 참고문서(zip) 다운로드하여 정확한 오퍼레이션명/파라미터 확인 필요.

### Step 4: 건물에너지사용량 API

| 항목 | 상태 |
|------|:----:|
| API 키 발급 + 승인 | **승인 완료** (즉시) |
| 에너지 사용량 (kWh) 반환 확인 | **대기중** (404 에러) |
| BD_MGT_SN **미포함** 확인 | ⬜ |
| 데이터 갱신 주기 (월별 집계) | ⬜ |
| 커버리지 (소규모 건물 데이터 유무) | ⬜ |

- **엔드포인트**: `https://apis.data.go.kr/1613000/BldEngyHubService` (오퍼레이션명 확인 필요)
- **ESG 활용**: 전기 사용량 × 탄소배출계수(0.4747 tCO2/MWh) → 간접 탄소 배출량
- **이슈**: 승인 완료 상태지만 404 반환. 참고문서(hwp) 다운로드하여 정확한 오퍼레이션명 확인 필요.

### Step 5: 카카오맵 API

| 항목 | 상태 |
|------|:----:|
| KakaoAK 인증 동작 | **PASS** |
| 좌표 → 지번주소 반환 | **PASS** (서울 중구 태평로1가 54-3) |
| 좌표 → 행정구역코드 반환 | **PASS** (법정동코드 1114010300) |
| 키워드 검색 (POI) | **PASS** (광화문교보빌딩 등 5건) |
| 무료 쿼터 30만건/일 (테스트 앱 100건/일) | 확인 |

- **엔드포인트**: `https://dapi.kakao.com/v2/local/`
- **역할**: 좌표↔주소 변환, POI 검색 (보조)
- **테스트 결과**: 좌표→주소, 좌표→행정구역코드, 키워드검색 모두 정상 동작.
- **참고**: 카카오맵 제품 설정에서 "카카오맵" 서비스 ON 필요 (초기 비활성화).

### Step 6: Google Cloud Vision API (OCR)

| 항목 | 상태 |
|------|:----:|
| API 키 발급 | **발급 완료** |
| 무료 1,000건/월 확인 | ⬜ |
| 한국어 간판 인식률 확인 | ⬜ |
| base64 이미지 → TEXT_DETECTION 동작 | **대기중** (결제 계정 필요) |

- **엔드포인트**: `https://vision.googleapis.com/v1/images:annotate`
- **역할**: 건물 간판 텍스트 인식 → 레이캐스팅 결과 교차 검증
- **이슈**: 무료 쿼터(1,000건/월)가 있지만 GCP 프로젝트에 결제 계정(billing) 연결 필요. 카드 등록 후 무료 범위 내 사용 가능.

## 핵심 발견: 데이터 연계 구조

### BD_MGT_SN은 만능 공통키가 아니다

초기 가설 (BD_MGT_SN으로 모든 API 연결) 은 **잘못됐다**.

실제 연계는 **PNU(필지고유번호)** 를 분해하여 각 API에 맞는 파라미터로 변환:

```
Vworld GIS건물통합정보 (BD_MGT_SN + PNU + 폴리곤)
       │
       ├── PNU (19자리) 분해
       │   ├── 시도코드 (2자리)
       │   ├── 시군구코드 (3자리)  ──→ sigungu_cd (5자리 = 시도+시군구)
       │   ├── 읍면동코드 (3자리)
       │   ├── 리코드 (2자리)     ──→ bjdong_cd (5자리 = 읍면동+리)
       │   ├── 토지구분 (1자리)
       │   ├── 본번 (4자리)       ──→ bun
       │   └── 부번 (4자리)       ──→ ji
       │
       ├──→ 실거래가 API     : LAWD_CD (시군구 5자리) + DEAL_YMD + 지번 매칭
       ├──→ 건축물대장 API   : sigunguCd + bjdongCd + bun + ji
       └──→ 건물에너지 API   : sigunguCd + bjdongCd + bun + ji

       └── BD_MGT_SN → 도로명주소 추출 → 카카오맵 보조 검증
```

### PNU 파싱 로직

```python
# PNU 19자리 구조:
# [시도(2)][시군구(3)][읍면동(3)][리(2)][토지구분(1)][본번(4)][부번(4)]

# 예시: 1111010100100010000
#   시도    = 11       (서울)
#   시군구   = 110      (종로구)
#   읍면동   = 101      (청운동)
#   리      = 00
#   토지구분 = 1        (대지)
#   본번    = 0001
#   부번    = 0000

# → sigungu_cd = "11110" (LAWD_CD로 사용)
# → bjdong_cd  = "10100"
# → 지번      = "1"
```

- **PNU 파싱 로직 구현 완료** — 19자리 → 시군구코드/법정동코드/번/지 자동 분해

### 아직 안 되는 것
- Vworld — 서비스URL 설정 문제로 인증 실패 (키 반영 대기 or 재발급)
- 건축물대장/에너지 — 건축HUB API 엔드포인트 오퍼레이션명 미확인 (참고문서 필요)
- Google OCR — GCP 결제 계정 미연결

### 주의사항
- 공공데이터포털 API는 XML 응답이 기본 (JSON 변환 로직 필요)
- API 키는 `.env`에 저장, git에는 올리지 않기 (`.gitignore` 설정 완료)
- Vworld 레이어명이 변경될 수 있으므로 여러 후보 시도 로직 필요
- 일일 1,000회 제한 → 캐싱 레이어 필수
- 카카오맵은 제품 설정에서 "카카오맵" 서비스 ON 필요 (초기 비활성화 상태)
